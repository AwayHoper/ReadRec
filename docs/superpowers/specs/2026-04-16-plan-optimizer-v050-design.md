# ReadRec v0.5.0 学习计划优化设计

## 背景

当前 ReadRec 的“学习计划”模块已经支持：

- 在“计划”页选择当前词库
- 配置每日词量
- 配置新词 / 复习词比例
- 保存当前词库对应的学习计划

但现状仍然偏工程化，主要问题有：

- 用户需要直接理解 `dailyWordCount`、`newWordRatio`、`reviewWordRatio` 这类底层字段
- 无法直观看到不同学习强度方案对完成周期的影响
- 无法快速从有限的几种推荐方案中做决策
- 每日抽词逻辑虽然已按新词/复习词比例抽取，但缺少“乱序”和“生词本加权”的明确产品化规则

本次 `v0.5.0` 的目标是把学习计划从“参数输入页”升级为“可理解、可预估、可快速选择的计划配置器”。

## 目标

本次版本聚焦两部分：

1. 优化学习计划模块
2. 优化每日学习抽词逻辑

明确不包含：

- 本次不做学习进度展示（签到版、进度）
- 本次不新增词库上传
- 本次不改动三轮学习流程的核心页面结构

## 范围

### 包含

- 比例调节器
- 方案列表
- 动态预计完成日期
- 固定四档新词强度方案
- 抽词乱序
- 按计划比例抽词
- 生词本复习词加权

### 不包含

- 学习进度看板
- 用户自定义方案档位
- 复杂统计分析
- 后台可配置权重系统

## 方案选择

本次采用“前端产品化配置 + 后端沿用现有计划字段”的轻量升级方案。

### 采用方案

- 前端展示比例调节器与方案列表
- 用户最终选择的是“某个推荐方案”
- 保存时仍写入当前已有的计划字段：
  - `dailyWordCount`
  - `newWordRatio`
  - `reviewWordRatio`
- 后端抽词逻辑继续围绕这三个字段运行，只增强抽词顺序与权重策略

### 这样做的原因

- 不需要新增 `StudyPlan` 表结构字段
- 不需要重做计划数据模型
- 能快速把产品交互升级，同时保持与现有学习流程兼容

### 不采用的方案

- 不把“方案档位”单独持久化为新的数据库字段
- 不做独立的“计划模板”实体
- 不引入按词库规模动态生成方案档位，本版固定四档

## 产品设计

## 3.1 学习计划

### A. 比例调节器

功能：

- 设置新词学习与旧词复习的数量配比

固定预设：

- `1:1`
- `1:2`
- `1:3`
- `1:4`

交互逻辑：

- 点击某个比例选项后，立即刷新下方“方案列表”
- 同时刷新“预计完成日期”
- 比例调节器本身不直接保存，只有当用户选中某个方案并点击保存后，才真正写入学习计划

### B. 方案列表

展示形式：

- 滚动式卡片列表
- 每一行代表一种学习强度方案

固定新词档位：

- `5`
- `10`
- `20`
- `40`

每个方案卡片包含：

- 新词数：每日新学单词数
- 复习数：根据当前比例自动计算出的每日复习词数
- 每日总量：新词数 + 复习数
- 需要天数：完成当前词库剩余新词所需天数

复习数计算规则：

- 若比例为 `1:n`
- 则 `复习数 = 新词数 * n`

每日总量计算规则：

- `每日总量 = 新词数 + 复习数`

需要天数计算规则：

- `需要天数 = ceil((总词数 - 已完成词数) / 每日新词数)`

交互逻辑：

- 点击某个方案卡片后，该方案成为当前选中计划
- 当前选中状态需要有明显高亮反馈
- 切换比例时，如果原先选中的方案档位仍然存在，则保留该档位，只重新计算它的复习数、天数与预测日期

### C. 动态预测

功能：

- 基于当前选中的方案，直观展示预计完成日期

计算公式：

- `预计完成日期 = 当前日期 + ceil((总词数 - 已完成词数) / 每日新词数)`

其中：

- `总词数`：当前词库总词数
- `已完成词数`：当前用户在该词库中的已学习词数
- `每日新词数`：当前选中方案的新词数

展示要求：

- 页面上以清晰文案展示：
  - 预计完成日期
  - 剩余需要天数

### D. 业务规则与交互细节

- 每个词库仍然对应一个独立学习计划
- 用户切换词库后，比例调节器、方案列表、动态预测都应基于该词库当前计划恢复
- 如果当前词库已有学习计划：
  - 页面要自动推导出当前属于哪个比例和哪个方案档位
- 如果当前词库没有学习计划：
  - 默认使用：
    - 比例 `1:1`
    - 新词档位 `10`
- 方案列表中的新词数固定为四档，不随词库规模动态变化
- 本次保留文章风格字段，但它不作为此版本的重点交互对象

### E. 名词定义

- 新词：词库中用户完全没进行过学习流程的单词
- 复习词：词已经进行过学习流程的单词
- 生词：经过一轮学习后用户选中的词
- 今日计划学习词：按照用户学习计划设定的新词与复习词比例组成的当日学习词集合

## 3.3 每日学习抽词逻辑

### 1. 抽词乱序

目标：

- 避免抽出来的单词集中在同一字母段，降低机械感

规则：

- 新词候选和复习词候选在筛选完成后，不能直接按数据库默认顺序取前 N 个
- 在最终生成当日单词列表前，需要进行乱序处理

实现要求：

- 乱序应发生在确定候选集合之后、写入当日 session 之前
- 乱序后仍需保留新词与复习词的目标数量约束

### 2. 按比例抽词

目标：

- 严格按照当前学习计划中的新词 / 复习词比例抽取当日单词

规则：

- 计划中的“新词数”来自方案卡片的新词档位
- 计划中的“复习数”来自当前比例推导结果
- 当复习词不足时：
  - 用剩余新词补齐
- 当新词不足时：
  - 用剩余复习词补齐

### 3. 生词本加权

目标：

- 让生词本中的词更有概率再次作为复习词出现

规则：

- 生词本词汇属于复习词候选集的高优先级部分
- 抽取复习词时，优先从生词本词汇中选取
- 若生词本候选不足，再从一般复习词中补齐

本次版本要求：

- 先实现“优先抽取”而不是复杂概率系统
- 即：在复习词候选排序中，生词本词排在前面，再做乱序与截取

## 前端设计

## 计划页

学习计划模块改为四个区域：

1. 当前词库摘要
2. 比例调节器
3. 方案列表
4. 动态预测与保存操作

### 页面行为

- 读取当前词库和当前词库的学习计划
- 从当前计划推导：
  - 当前比例选项
  - 当前方案档位
- 当用户切换比例时：
  - 方案列表实时重算
  - 动态预测实时刷新
- 当用户切换方案时：
  - 动态预测实时刷新
- 点击保存后，把当前选中的结果转换为现有计划字段提交给后端

## 后端设计

### StudyPlan 持久化

数据库字段保持不变：

- `dailyWordCount`
- `newWordRatio`
- `reviewWordRatio`
- `articleStyle`

前端与后端约定：

- `dailyWordCount` 存“每日总量”
- `newWordRatio` 存比例左侧数值，本版本固定为 `1`
- `reviewWordRatio` 存比例右侧数值，即 `1 / 2 / 3 / 4`

例如：

- 若用户选择比例 `1:3`，新词档位 `10`
- 则：
  - `newWordRatio = 1`
  - `reviewWordRatio = 3`
  - `dailyWordCount = 40`

### 抽词逻辑

后端抽词时应改为：

1. 根据 `newWordRatio : reviewWordRatio` 计算目标新词数与目标复习词数
2. 从新词候选中抽取目标新词数
3. 从“生词本优先”的复习词候选中抽取目标复习词数
4. 若一侧不足则由另一侧补齐
5. 对最终结果进行乱序
6. 写入当日 session

### 已完成词数来源

动态预测和抽词规则中的“已完成词数”统一使用当前词库的已学习词数，即：

- `UserBookProgress.learnedWordIds.length`

## 数据流

1. 用户进入“计划”页
2. 页面读取当前词库、当前学习计划、当前词库进度
3. 页面基于当前词库总词数和已学习词数生成四档方案
4. 用户切换比例
5. 页面重新计算每档方案的复习数、总量、剩余天数和预计完成日期
6. 用户选中某一档方案
7. 页面将该方案映射成现有 `StudyPlan` 字段
8. 用户保存计划
9. 次日生成学习 session 时，后端按新的抽词规则生成当日学习词

## 错误处理

- 若词库信息或计划信息加载失败，计划页显示错误提示并禁止保存
- 若当前词库没有学习记录，已完成词数按 `0` 处理
- 若剩余新词数为 `0`，预计完成日期显示为“已完成当前词库”
- 若方案无法完整满足比例要求，页面不报错，只在后端抽词时按补齐规则处理

## 验证标准

- 比例调节器仅展示 `1:1 / 1:2 / 1:3 / 1:4`
- 方案列表固定展示 `5 / 10 / 20 / 40` 四档新词方案
- 切换比例后，方案列表中的复习数、总量、需要天数实时变化
- 选中方案后，动态预测实时显示预计完成日期
- 保存后重新进入页面，能恢复当前比例和当前方案档位
- 抽词时能按当前比例抽取新词和复习词
- 抽词结果不是按字母顺序集中排列
- 生词本中的词在复习词抽取中优先级更高
